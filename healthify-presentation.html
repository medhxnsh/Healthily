01 — Problem Statement

Blood reports are written for clinicians, not patients.

01

Opaque numbers

Values like "MCH: 24.3 pg" mean nothing to a patient. No context, no severity, no next step.

02

Range flags without meaning

A "Low" flag tells you nothing about what condition it might indicate or how worried to be.

03

Siloed interpretation

Doctors correlate 20+ parameters together. Patients see individual numbers with no pattern view.

04

No accessible tools

Premium health apps charge for AI interpretation. There is no free, intelligent, multi-format alternative.

02 — Solution

One upload. Six layers of analysis.

  • Multi-format ingestion — CSV, PDF, JPG, PNG, WEBP up to 10MB. No manual entry required.
  • Plain-English AI summary — Groq Llama 3.3 70B translates clinical values into patient language.
  • Z-Score anomaly scoring — Quantifies how far each value is from the healthy reference range.
  • Isolation Forest detection — ML detects unusual patterns across all parameters together.
  • Symptom-aware risk assessment — Maps blood values + reported symptoms to 15 condition probabilities.
  • SHAP explainability — Shows exactly which parameters drove the ML prediction.

06 — AI Layer

Two Llama models. Two different jobs.

MODEL 01

llama-3.3-70b-versatile

Simplifier — turns a table of lab values into a plain-English paragraph a patient can read and act on.

  • Receives: parameter name, value, unit, status (low/normal/high/critical)
  • Receives: patient age and biological sex for context
  • Returns: 2-4 sentence natural language summary
  • Prompt: explicitly prohibits medical advice language
  • Cached: identical parameter sets reuse prior summary
MODEL 02

llama-4-scout-17b-16e-instruct

Vision parser — reads blood report images (JPG/PNG/WEBP) and extracts structured parameter data without OCR libraries.

  • Receives: base64-encoded image at full resolution
  • Returns: JSON array of { name, value, unit, raw_name }
  • Handles: printed tables, handwritten values, non-standard formats
  • Fallback: parse errors surface to user with clear message
  • Post-process: canonical name mapping + physical limit validation
Powered by Groq API  ·  OpenAI-compatible endpoint  ·  https://api.groq.com/openai/v1

07 — Machine Learning Models

Four components. Each solves a different problem.

XGBOOST CLASSIFIER · xgboost

Condition Classifier

Multi-class robust anomaly classifier trained on 3,918 class-balanced rows across 6 clinical datasets. SMOTE scaling solves class imbalances and boosts accuracy to 95%.

OUTPUT → 15-class probability distribution · top condition + confidence
ISOLATION FOREST · scikit-learn

Multivariate Anomaly Detector

100 isolation trees, contamination=0.1. Detects unusual patterns across all blood values simultaneously — catches cases where individual values look normal but combined pattern is abnormal.

OUTPUT → anomaly score (−1 to +1) per sample · flag per parameter
Z-SCORE · statistical

Per-Parameter Outlier Scorer

Computes (value − reference_mean) / reference_std using age- and sex-adjusted reference ranges from reference_ranges.json. Threshold: |z| > 2.0 flagged. Direction (high/low) returned.

OUTPUT → float z-score + direction + is_anomaly bool per parameter
SHAP · TreeExplainer

Feature Importance Explainer

SHapley Additive exPlanations assigned to each feature for the top predicted condition. Positive SHAP = increased prediction probability. Negative = decreased. Visualised as horizontal bar chart.

OUTPUT → SHAP value + direction per feature · top contributing parameters

11 — Anomaly Detection

Statistical outliers AND pattern-level anomalies.

Method 01 — Statistical

Z-Score Analysis

Measures how many standard deviations each value sits from the reference mean. Uses age- and sex-adjusted ranges.

−3
+3
  • Score near 0 = value is average for healthy population
  • ±1 = slightly above/below average, normal range
  • ±2 = flagged — worth monitoring
  • ±3 = significantly outside normal, see a doctor
  • Direction returned: HIGH or LOW with each score
Method 02 — Machine Learning

Isolation Forest

Builds 100 isolation trees. Anomalous samples are isolated in fewer splits. Looks at ALL parameters together — not one at a time.

−1
+1
  • Score near +1.0 = pattern matches healthy population
  • Score near −1.0 = pattern is highly unusual
  • Catches cases where individual values look OK but combined pattern is wrong
  • contamination = 0.1 (10% expected anomaly rate)
  • 100 estimators, random_state=42

12 — Risk Assessment Engine

Symptoms + blood values → condition risk percentages.

The risk engine maps user-reported symptoms and out-of-range blood values to a set of condition risk scores using a rule-based weighted scoring system backed by symptom_condition_map.json and safety_conditions.json.

Each condition gets a score between 0–100% based on how many of its associated symptoms and lab abnormalities are present. Conditions with risk > 0% are shown. The requires_doctor flag is surfaced when risk is non-zero.

45+ symptoms supported — formatted from SNAKE_CASE to human-readable labels in the UI.

EXAMPLE OUTPUT

Iron Deficiency Anemia 74%
Hypothyroidism 41%
Type 2 Diabetes 28%
Liver Disease 12%

13 — SHAP Explainability

Why did the model predict that? SHAP shows you.

SHAP (SHapley Additive exPlanations) assigns a contribution value to each blood parameter for the predicted condition. Based on game theory — each feature's "fair share" of the prediction is calculated.

Crimson bars (positive SHAP) — this parameter increased the predicted condition's probability.

Blue bars (negative SHAP) — this parameter decreased the probability.

Bar length = magnitude of influence. Helps patients understand which specific lab values drove the ML result, not just what was predicted.

15 — Engineering Challenge

Why the model always predicted microcytic anemia — and how we fixed it.

Root Cause

Missing features were filled with sick-population medians computed from the imbalanced training data. Every absent parameter silently pushed predictions toward anemia.

# Training dataset medians (skewed) hemoglobin: 12.3 # below normal — anemia range bun: 41.5 # 3× normal max alp: 208.0 # 3× normal albumin: 3.1 # below normal creatinine: 1.3 # elevated # Result: every CBC-only report looked like # a sick patient with anemia + kidney issues
Fix Applied

Added _HEALTHY_DEFAULTS dictionary in classifier.py. Missing features now use physiologically normal reference values — no retraining required.

_HEALTHY_DEFAULTS = { "hemoglobin": 14.0, # normal midpoint "bun": 14.0, # normal "alp": 70.0, # normal "albumin": 4.2, # normal "creatinine": 0.9, # normal # ... all 21 features } # Result: absent features assumed healthy # Prediction only reflects what's in the report

16 — Limitations & Roadmap

Where we are. Where we're going.

Current Limitations

  • NOW Siloed datasets — each covers only one panel type, no patient has a full multi-panel profile in training data.
  • NOW ~6,800 training rows is modest. CKD has only 400 rows — probability estimates may be unreliable.
  • NOW Image parsing quality depends on Groq Llama 4 Scout — non-standard or low-res formats may extract incorrectly.
  • NOW Single-snapshot analysis only. No longitudinal trend tracking across multiple reports.
  • NOW No clinical validation — the model has not been tested against confirmed diagnoses from real patients.

Planned Improvements

  • DONE XGBoost integration — Handles non-linear relationships. We achieved 95% test accuracy.
  • DONE Derived ratio features — BUN/Creatinine, AST/ALT, MCH/MCV safely tracked and fed to XGBoost.
  • DONE Unit normalisation — Implemented a robust sub-string standardizer that rescales shorthand clinical features.
  • NEXT MIMIC-IV integration — millions of real ICU lab results with ICD-10 diagnoses. Full multi-panel per patient.
  • FUTURE Longitudinal tracking — trend charts across multiple reports, flag parameters crossing normal/abnormal boundaries.

Blood reports should be readable by everyone.
